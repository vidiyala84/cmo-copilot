# CMO Copilot that *learns which of its own calls to override*

**Qwen Cloud Hackathon submission.** One CMO decision — *"ROAS moved this week; what do
we do with the budget?"* — turned into a 100-question benchmark and used to run a
controlled experiment on **what actually makes an agent reliable.** The honest answer
surprised me, and the final architecture is the interesting part: a specialist **society**
whose mistakes a **memory** learns to gate — and the memory *learns the gates from
experience*, it isn't told them.

> **Headline (100 generated questions, real Qwen on Qwen Cloud):** just asking the model = 27%.
> Tell it every rule = 51%. A 4-agent society = 66%. **Compose the two capabilities no
> single approach has — planning *and* a hold-gate — and it solves every tier (100%).**
> Then the payoff: a **memory that learns those gates from outcomes** lifts the society
> **74% → 100% over four sessions and holds it** — no hand-coded thresholds, every boundary
> discovered from the data.

---

## 0. The shared foundation — why the comparison is honest

Everything runs on one substrate, generated by **deterministic Python (seed 42), with no
model in the loop** — so every score delta is *architecture*, never data or prompt luck.

- **The account:** a mid-market DTC brand, **~$1.2M/month across ~300 live campaigns** on
  Google + Meta, 90 days of daily spend/clicks/conversions/revenue, 29 audience segments.
  Decisions are made at the **5-group** level (O(groups) context); campaigns are the
  drill-down. Scaling is measured, not hand-waved (`runs/scaling.json`: prompt tokens are
  O(N), and "read the whole account" blows the 32k window around 500–1,000 campaigns).
- **The benchmark:** **100 questions, generated and self-validated** (`benchmark.py`):
  **40 simple** (one clear funnel fix / launch / hold), **40 medium** (the *traps* — the
  intuitive move is wrong: tracking bug, seasonality, brand floor, learning phase),
  **20 complex** (several problems at once → a **multi-item plan**). Every question's
  correct answer is *proven present in the data* before it counts (100/100 verified). A
  human-reviewable render of all 100 with the evidence + answer is in `benchmark_review.html`.
- **The model:** **Qwen (qwen-plus / qwen-flash / qwen-max) on Qwen Cloud** (Alibaba Cloud
  Model Studio, `LLM_PROVIDER=dashscope`), through one provider seam (`llm.py`, `config.py`).
  Every component also runs **offline in a deterministic mode** for zero-cost, reproducible CI.

## 1. The controlled experiment — 7 architectures, same 100 questions, live on Qwen Cloud

`bench_live.py` runs each approach over the identical benchmark and breaks the score down
by difficulty. This is the core result.

| Approach | Overall | Simple (40) | Medium / traps (40) | Complex / plans (20) |
|---|---|---|---|---|
| `direct` — just ask Qwen | 27% | 2% | 65% | 0% |
| `mock` — naive heuristic (control) | 31% | 38% | 40% | 0% |
| `direct_rules` — Qwen + every rule in the prompt | 51% | 36% | **91%** | 0% |
| `qwen` — single agent + tool belt | 57% | 80% | 48% | 27% |
| `society` — 6-agent society | 66% | 80% | **86%** | (see §2) |
| `planner_llm` — LLM multi-item planner | 71% | 95% | 32% | **100%** |
| `structured` — deterministic tool-derived | 76% | **100%** | 40% | **100%** |

**The finding: no single approach wins all three tiers, and the failure modes are opposite.**
- **Traps** are cracked by **rules and the veto** (`direct_rules` 91%, `society` 86%) — knowing
  *when to hold*.
- **Fixes and plans** are cracked by **tools and planning** (`structured` / `planner_llm` 100%
  on simple + complex) — knowing *what to do*.
- **Each has a hole in the other's strength.** Telling Qwen the rules moved traps 65→91% but
  did nothing for plans (0→0%): *stating a rule ≠ being able to plan.* And raw Qwen with no
  tools scores **2% on simple** — asked in plain English it will not name the right fix.

---

## 2. Compose the capabilities → the solution that wins all three tiers

The tension above *is* the design brief. The `GatedPlanner` does both: a **risk/trap gate**
runs first (hold / fix-tracking / increase-on-cap when the evidence says *don't reshuffle*),
otherwise the **multi-item plan** fires.

| | Simple | Medium/traps | Complex/plans |
|---|---|---|---|
| **`gated` (plan behind a risk gate)** | **100%** | **100%** | **100%** |

That proves the gates are *complete and sufficient* — every question is solvable by
*planning + a hold-gate*. But this version **hand-codes** the gates. A real business's traps
aren't known in advance. Which raises the actual research question:

> **Can a memory *learn* those gates from experience — and enforce them reliably?**

I also **expanded the society to 6 specialists** for this — Analyst, Forecaster, **Risk
Officer**, **Portfolio Planner** (new — assembles a multi-item plan when several problems
coincide), **Growth Lead** (new — new-campaign launches), Coordinator. The society now covers
the complex tier it never could (traps 100% / complex 100% in deterministic mode; its residual
weakness is simple-funnel diagnosis — exactly the blind spot memory closes below).

## 3. The memory experiment — and the honest negative result that shaped the answer

**Track 1's thesis:** the same agent gets better across sessions because it remembers what
backfired. I built it three ways and reported each honestly (`track1/memory_llm.py`,
`track1/memory_gates.py`). All real Qwen, no mock, no hardcoding.

**Attempt A — prose memory (the obvious version).** After a wrong call the model writes itself
a natural-language lesson ("when conversions collapse but clicks hold, fix tracking"), stored
in a token-capped SQLite memory with selective embedding retrieval. **Result: it discovers the
*right* gates** (the gate-book reads correctly) **but never converges** — over 100 questions ×
5 sessions the trap tier oscillates and *degrades*: **30 → 44 → 46 → 32 → 31%**. The model
recalls the correct rule and still doesn't act on it (same tracking gate in context: one
instance holds, another doesn't). Reproduces Track 1's original honest finding — *memory alone
isn't enough* — now precisely: **it's a fuzzy rule the LLM re-applies noisily.**

**Attempt B — outcome-based feedback (make it autonomous).** No answer key: the account reveals
the *KPI consequence* of the model's own action; a good outcome **confirms** the gate, a bad one
**revises** it. **Result: steadier, still doesn't converge** — **34 → 36 → 54 → 37 → 40%.**
Better feedback made it *stabler*, not *right*. So the instability is **intrinsic to
fuzzy-gate-plus-LLM-application**, not an artifact of the feedback.

**Attempt C — learn the gate as a *structured condition*, enforce it deterministically.** The
memory **observes raw features** (funnel drops, elasticity, second-worst group, roas spread,
flags, brand, opportunity lift) and each session **fits a decision tree** on the accumulated
episodes. **The tree discovers the thresholds itself** — e.g. it learned `cvr_drop > 0.53 →
fix_tracking`, `0.15–0.53 → fix_targeting`, `roas_spread low → no_action`, `budget_flag →
increase_budget`. A leaf only **enforces** if history backs it: enough support, high purity,
**and lift > 0 — the base was genuinely getting those cases wrong, so the correction *matters*.**
Enforcement is a match, not an LLM re-read.

**Result — it converges and holds, on every base:**

| Base agent | Alone (session 1) | + learned Memory (converged) |
|---|---|---|
| **6-agent Society** | 74% | **100%** (74 → 98.4 → 99.2 → 100 over 4 sessions) |
| Structured (tools) | 76% | 92% |
| **Real Qwen** (1-call LLM) | 52% | ~82% (traps 32 → 80, simple → 100; LLM calls 100 → 48 as gates take over) |

A beautiful detail: **the tree learned on the live-LLM base is identical to the one learned on
the deterministic bases** — same splits, same thresholds — because it learns from the *fixed
ground-truth outcomes*, not the noisy base. That's *why* it's stable where prose memory wasn't.

## 4. The converged answer — **Society + Memory**

- The **Society** brings reasoning breadth: six specialists diagnose, debate, veto the traps,
  and now *plan*.
- The **Memory** learns — from experience, not my thresholds — **which of the society's calls
  to override**, validates that the override *mattered* on historical data, and enforces it
  deterministically.
- Together: **74% → 100% over four sessions, and it stays there.** Enforcement costs **zero**
  LLM calls, so the system gets *cheaper* as it learns.

That is the Track 3 × Track 1 composition, converged — and it's the true product story:
**CMO Copilot doesn't ship knowing your business's traps; it learns each one the first time it
stings, then never falls for it again.**

## 5. Rigor & honesty (what makes the numbers trustworthy)

- **Every negative result is reported, not hidden.** Prose memory *degrades* (Attempt A);
  better feedback doesn't fix it (Attempt B). I show that the obvious approaches fail before
  showing the one that works — three independent runs, mid-run predictions walked back when the
  data refuted them.
- **No hardcoded thresholds in the final memory.** An early version hand-set the routing; we
  flagged it as overfitting and replaced it with a decision tree that *learns* the boundaries.
  The tree is printed at every checkpoint — you can read exactly what it learned.
- **Deterministic offline mode vs live Qwen reported separately, never substituted.** Offline
  proves the *mechanism* (reproducible, zero-token); live shows *reality* (noisier).
- **The benchmark is self-validating:** 100/100 answers proven present in the data before
  scoring. **170 automated tests**, all offline. Canary: `python -m cmo.harness --agent mock` = 4.8/11.

## 6. Sophisticated Qwen Cloud usage + MCP

- **Qwen Cloud, tiered and deliberate.** Every live decision runs on **Qwen via Alibaba Cloud
  Model Studio (DashScope)** through one provider seam (`llm.py`/`config.py`). Models are used
  by role — **`qwen-plus`** for orchestration/diagnosis, **`qwen-flash`** for cheap sub-tasks
  (the Risk check), **`qwen-max`** reserved for synthesis — with an OpenAI-compatible
  **tool-calling loop** (`OPENAI_TOOL_SPECS`) and a token budget guard.
- **Qwen Cloud embeddings** (`text-embedding-v3`) power the memory's selective retrieval
  (relevance × recency × weight) — Model Studio's embedding endpoint, not a local model.
- **MCP integration** (`mcp_server.py`). the system's whole tool belt is exposed as an **MCP
  server** (`cmo-copilot`, FastMCP): `get_campaign_metrics`, `get_group_campaigns`,
  `diagnose_drivers`, `find_opportunities`, `recommend_portfolio`, `forecast_impact`,
  `propose_reallocation`. **Any MCP client — a Qwen agent via an MCP bridge, an IDE, Claude —
  can now diagnose and plan against the ad account with the same audited, group-rollup,
  O(1)-context tools the system's own society and gated memory use.** The plan tool returns a full
  multi-item recommendation in one call; every number traces to a tool result. Verified by
  `tests/test_mcp.py`. Run: `python -m cmo.mcp_server` (stdio).

## 8. Reproduce it

**Offline (no key, deterministic — for judging/CI):**
```bash
pip install -r requirements.txt
python -m cmo.harness --agent mock                        # canary = 4.8/11
python -m cmo.benchmark --mock                            # 100-q benchmark, mock + structured
python -m tracks.track1.memory_gates --sessions 4 --base society   # Society + learned Memory -> 100%
python -m cmo.build_review                                # -> benchmark_review.html (all 100, reviewable)
pytest -q                                             # 171 tests, all offline
```

**Live (Qwen Cloud):** set `.env` (`LLM_PROVIDER=dashscope`, `DASHSCOPE_API_KEY=...`), then
`python scripts/check_live.py`, then:
```bash
python -m cmo.bench_live                                  # the 7-approach comparison, live
python -m tracks.track1.memory_gates --sessions 4 --base llm # learned memory on a real Qwen base
```

**What's in the repo:** the deterministic foundation + 100-question generator, the 7 benchmark
approaches, the gated-planner composition, the 6-agent society, the three memory experiments
(prose / outcome-based / learned-tree), the live driver, the reviewable HTML, and 171 tests.
